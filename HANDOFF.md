# Persona Pipeline Handoff Guide

Welcome! This guide will help you understand and run the persona analysis pipeline. If you're reading this, you're probably taking over maintenance of the buyer persona intelligence system that helps our Marketing and Sales teams understand who they're talking to.

---

## 1. What This Pipeline Does

**The Problem**: Our Sales and CS teams talk to hundreds of different buyer personas across different company sizes (Digital, SMB, Commercial, Enterprise). Each persona has different pain points, goals, objections, and success metrics. Without structured intelligence, reps waste time figuring out what resonates with each role.

**What It Takes as Input**: The pipeline pulls the last 6 months of Gong call transcripts from Snowflake (~79,000+ calls). Each call includes metadata like customer job title, company segment, call date, and AI-generated call summaries (key points, brief, spotlight data).

**What It Produces**: 
1. **Team Review Page** - An interactive HTML page where internal reviewers can vote on insights, add comments, and edit persona data inline
2. **Final Stakeholder Page** - A clean, polished page published to GitHub Pages for Sales/CS teams to reference during calls
3. **Persona Insights JSON** - Structured data with pain points, goals, objections, success metrics, messaging preferences, and competitive landscape for 15 personas across 4 segments
4. **Pipeline Metadata** - Tracking info about what data was analyzed when

**How Often to Run**: Quarterly (once per quarter). More frequent runs are possible but expensive due to Claude API costs. The pipeline costs approximately $150-200 per full run in API fees.

## 3. Setup Instructions

### Prerequisites
- Python 3.9 or higher
- Git
- GitHub account with access to the `chrissherman-png/persona-analysis` repository
- Zendesk email address (for Snowflake SSO authentication)
- Access to Zendesk's Snowflake data warehouse
- Access to Zendesk's Claude API gateway (AI Gateway)

### Step 1: Clone the Repository

```bash
git clone https://github.com/chrissherman-png/persona-analysis.git
cd persona-analysis
```

### Step 2: Create Environment File

```bash
cp .env.example .env
```

Then edit `.env` and add your Zendesk email:

```bash
# .env file
SNOWFLAKE_USER=your.email@zendesk.com

# Claude API credentials (should already be set)
AWS_ENDPOINT_URL_BEDROCK_RUNTIME=https://ai-gateway.zende.sk/bedrock
AWS_BEARER_TOKEN_BEDROCK=<ask your team for this>
ANTHROPIC_MODEL=us.anthropic.claude-sonnet-4-5-20250929-v1:0
```

**Important**: Never commit your `.env` file to git. It contains credentials and is automatically ignored by `.gitignore`.

### Step 3: Install Python Dependencies

The easiest way is to use a virtual environment:

```bash
# Create virtual environment
python3 -m venv venv

# Activate it
source venv/bin/activate  # On macOS/Linux
# OR
venv\Scripts\activate     # On Windows

# Install dependencies
pip install snowflake-connector-python pandas anthropic python-dotenv httpx boto3
```

If you prefer to install globally (not recommended):

```bash
pip3 install snowflake-connector-python pandas anthropic python-dotenv httpx boto3
```

### Step 4: Test Your Connection (Optional)

```bash
python3 test_connection.py
```

This will verify your Snowflake credentials work before running the full pipeline.

### Step 5: Start the Flask Server (Optional)

The Flask server is only needed if you want to make manual inline edits to persona data outside of the Team Review page workflow.

```bash
python3 simple_server.py
```

Server runs at `http://localhost:8080` and provides a UI for editing JSON files directly.

## 4. Running the Pipeline

### Full Pipeline Run

From the repo directory:

```bash
python3 run_pipeline.py
```

### What Happens Step-by-Step

**Step 1: Snowflake Extraction (5-10 minutes)**
- Your browser will automatically open for Okta SSO login
- Log in with your Zendesk credentials
- The script pulls the last 6 months of Gong calls
- Saves to `data/gong_calls_YYYYMMDD_HHMMSS.csv` (2+ GB file)
- Logs total calls extracted (typically ~79,000+)

**Step 2: Claude Analysis (25-35 minutes)**
- Loads the CSV and applies filters:
  - English language only (excludes Italian/German calls)
  - Customer speakers only (excludes Zendesk rep speech)
  - Assigns personas based on job titles
- For each persona in each segment:
  - Samples ~400-1700 customer statements from calls
  - Sends to Claude API for analysis
  - Extracts: pain points, goals, objections, success metrics, messaging preferences, product requirements, information sources, competitive mentions
- Caches results to `cache/` directory for faster re-runs
- Logs progress: "Analyzing: CX Champion in SMB..."

**Step 3: Generate Team Review Page (1-2 minutes)**
- Combines Claude insights with any pending edits from prior reviews
- Generates interactive HTML page with voting, commenting, and inline editing features
- Saves to `reports/Persona_Team_Review_Full.html`

### Expected Output

After a successful run, you'll see:

```

📋 Next steps:
```

## 5. Understanding the Output

### Team Review Page (`Persona_Team_Review_Full.html`)

**Purpose**: Internal collaborative review interface

**Features**:
- **Voting**: Thumbs up/down on each insight (tracked in Google Sheets)
- **Commenting**: Add notes or questions on any section
- **Inline Editing**: Click any text to edit directly in the page
- **Change Tracking**: All edits are saved to `pending_changes/edits_Q2_2026.json`
- **Version History**: Snapshots saved to `versions/personas_YYYY-MM-DD.json`

**Who Uses It**: Data scientists, sales ops, CS leadership reviewing the AI-generated insights before they go live

**Where Edits Are Saved**:
- **Votes & Comments**: Google Sheets (configured in the page header)
- **Text Edits**: `pending_changes/edits_Q2_2026.json` (local file)

**Important**: This page contains RAW insights that may need editing. Do not share externally until reviewed.

### Final Stakeholder Page (`Persona_Profiles_FINAL.html`)

**Purpose**: Clean, stakeholder-facing persona profiles

**Features**:
- No voting or editing UI (read-only)
- Polished content with edits applied
- Organized by segment → persona
- Includes: job titles, pain points, goals, objections, success metrics, messaging guidance

**Who Uses It**: Sales reps, CS teams, product managers preparing for customer calls

**How It's Generated**: Run this after Team Review edits are complete:

```bash
python3 generate_final_clean_personas.py
```

This reads `pending_changes/edits_Q2_2026.json` and applies all approved edits to generate the final page.

**Publishing**: Push to GitHub Pages the same way as Team Review:

## 7. Key Things to Know

### ⚠️ English Language Filter Fix (June 2026)

**Critical**: The pipeline includes a corrected English language filter that was fixed in June 2026. 

**What Was Broken**: The old filter used substring matching (`if word in text_lower`) which caused massive false positives:
- "questions" matched the Italian word "questi" → incorrectly rejected
- "channel" matched the Italian word "nel" → incorrectly rejected
- "database" matched the Italian word "base" → incorrectly rejected

This caused **79% of valid English calls to be incorrectly rejected as Italian**.

**What Was Fixed**: The filter now uses regex word boundaries (`\b` markers) to match only complete words:
- "questions" no longer matches "questi" ✓
- "channel" no longer matches "nel" ✓
- "database" no longer matches "base" ✓

**Result**: Pass rate increased from 12.8% to 82.1% (6.4× more calls analyzed).

**Impact for You**: 
- If you re-run the pipeline with cache cleared, you'll analyze **54,972 additional calls** that were previously incorrectly rejected
- This is expected behavior, not a bug
- Do NOT revert to the old filter logic

### Cache Management

Claude API responses are cached in `cache/` directory to avoid re-analyzing the same data multiple times.

**When to Clear Cache**:
- You're adding new personas or changing persona definitions
- You're updating the Claude prompt to extract different insights
- You want to analyze completely fresh data after a Gong data refresh

**How to Clear Cache**:
```bash
rm -f cache/*.json
```

**When NOT to Clear Cache**:
- Just re-running the pipeline to test page generation
- Applying edits from the Team Review page
- Updating HTML templates or styling

**Why Cache Matters**: A full Claude analysis run costs $150-200 in API fees. Using cache saves money and time (3 minutes vs. 30 minutes).

### Snowflake Authentication

The pipeline uses Snowflake's "externalbrowser" authentication (SSO via Okta).

**First Run**:
- Browser window opens automatically
- Log in with your Zendesk Okta credentials
- Snowflake caches your session token

**Subsequent Runs**:
- If your session token is still valid, no browser prompt
- If expired, browser opens again for re-authentication

**Troubleshooting Auth**:
If auth fails or hangs:
```bash
# Clear Snowflake cached credentials
rm -rf ~/.snowflake
```

Then run the pipeline again.

### GitHub Pages Publishing

The Final page is automatically published to GitHub Pages when you push to `main` branch.

**How It Works**:
- GitHub Pages is configured to serve from the `main` branch, `/` directory
- Any HTML files in `reports/` are accessible at:
  ```
  https://chrissherman-png.github.io/persona-analysis/reports/<filename>.html
  ```
- Updates appear live within 1-2 minutes after pushing

**What Gets Published**:
- ✓ `reports/Persona_Team_Review_Full.html` (if you push it)
- ✓ `reports/Persona_Profiles_FINAL.html` (if you push it)
- ✗ `data/` directory (gitignored, too large)
- ✗ `cache/` directory (gitignored)

**Security Note**: GitHub Pages sites are **public by default**. Anyone with the URL can access the pages. If you need to restrict access:
1. Use a private GitHub repository (requires GitHub Enterprise)
2. Or host the pages on an internal server instead of GitHub Pages

### Cost Considerations

**Claude API Costs**: ~$150-200 per full pipeline run
- Cost scales with number of calls analyzed
- Cached runs cost $0 (no API calls made)
- Consider budget before clearing cache and re-running

**Snowflake Costs**: Minimal
- Query execution time: ~5-10 minutes
- Data transfer: ~2 GB
- Typical cost: <$5 per run

## 8. Troubleshooting

### Problem: `"SNOWFLAKE_USER not set"` error

**Symptoms**: Pipeline exits immediately with environment variable error

**Solution**:
1. Check that `.env` file exists in the repo root:
   ```bash
   ls -la .env
   ```
2. Open `.env` and verify it contains:
   ```bash
   SNOWFLAKE_USER=your.email@zendesk.com
   ```
3. Make sure there are no extra spaces or quotes around the value
4. Try running again:
   ```bash
   python3 run_pipeline.py
   ```

---

### Problem: Pipeline times out or hangs after 30 seconds

**Symptoms**: Script starts but stops responding

**Possible Causes**:
1. **Snowflake auth waiting for browser**: Check if a browser window opened but you didn't log in
2. **Claude API rate limiting**: Too many requests in a short time

**Solutions**:
- **For Snowflake auth**: Look for a browser window that opened in the background. Log in to Okta and authorize Snowflake.
- **For rate limiting**: Wait 5-10 minutes and try again. The Claude API has rate limits that reset over time.

---

### Problem: `"No pending changes to apply"` message

**Symptoms**: When running `generate_final_clean_personas.py`, you see "No pending changes to apply"

**Explanation**: This is **not an error**. It means no one has made inline edits in the Team Review page yet.

**Solutions**:
- **If you want to generate the final page without edits**: This is fine. The script will generate the page using the base Claude insights.
- **If you expected edits to be applied**: Check that edits were actually saved to `pending_changes/edits_Q2_2026.json`. Open that file and verify it contains edits.

---

### Problem: Team Review page doesn't show newest data

**Symptoms**: You ran the pipeline but the Team Review page still shows old insights

**Cause**: The page is generated from cached results, not fresh data

**Solution**:
1. Clear the cache:
   ```bash
   rm -f cache/*.json
   ```
2. Re-run the pipeline:
   ```bash
   python3 run_pipeline.py
   ```

This forces fresh Claude analysis of all personas.

---

### Problem: Git push fails with authentication error

**Symptoms**: `git push origin main` fails with "Permission denied" or "Authentication failed"

**Cause**: Your GitHub credentials are not configured or expired

**Solution**:

**For HTTPS URLs**:
```bash
# Use macOS keychain for credential storage
git config --global credential.helper osxkeychain

# Try pushing again
git push origin main
```

If that doesn't work, you may need to generate a Personal Access Token (PAT):
1. Go to GitHub → Settings → Developer settings → Personal access tokens → Tokens (classic)
2. Generate new token with `repo` scope
3. Use the token as your password when prompted

**For SSH URLs**:
```bash
# Check if your SSH key is added to GitHub
ssh -T git@github.com

# If it fails, add your SSH key
# Generate one if you don't have it:
ssh-keygen -t ed25519 -C "your.email@zendesk.com"

# Add to ssh-agent
ssh-add ~/.ssh/id_ed25519

# Add the public key to GitHub
cat ~/.ssh/id_ed25519.pub
# Copy output and paste into GitHub → Settings → SSH and GPG keys
```

---

### Problem: "Failed to parse Claude response as JSON" error

**Symptoms**: Pipeline crashes during Claude analysis phase with JSON parse error

**Cause**: Claude's response was truncated or malformed (rare but possible)

**Solution**:
1. Check the log for which persona failed
2. Delete just that persona's cache file:
   ```bash
   # Example: if "SMB_CX_Champion" failed
   rm cache/SMB_CX_Champion.json
   ```
3. Re-run the pipeline (it will re-analyze just that persona)

If it fails repeatedly on the same persona, there may be an issue with the prompt or the data for that persona. Reach out to the team for help.

---

### Problem: `ModuleNotFoundError: No module named 'snowflake'`

**Symptoms**: Python can't find Snowflake connector

**Solution**:
```bash
# If using venv
source venv/bin/activate
pip install snowflake-connector-python

# If not using venv
pip3 install snowflake-connector-python
```

Then try running the pipeline again.

---

### Problem: Cache directory doesn't exist

**Symptoms**: `FileNotFoundError: [Errno 2] No such file or directory: 'cache/'`

**Solution**:
The pipeline should create `cache/` automatically, but if it doesn't:
```bash
mkdir -p cache
```

Then re-run the pipeline.
